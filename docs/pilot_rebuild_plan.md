# Пилот MAX Case Bot: аудит и план перестройки

## 1. Что требует ТЗ

Пилот должен уметь:

- принимать свободное описание производственной ситуации;
- подбирать 1-3 релевантных кейса;
- показывать карточку кейса;
- вести пользователя по шагам внутри диалога;
- принимать комментарии и фото по шагам;
- сохранять историю прохождения;
- принимать заявку на новый кейс;
- давать минимальный административный контур и аналитику.

Критический фокус MVP:

- `P0`: живой бот в MAX, поиск, карточка, прохождение, создание нового кейса, базовые события;
- `P1`: подсказки по шагам, популярные/последние кейсы, базовый отчет, публикация новых кейсов;
- `P2`: расширение хранилища, админка, зрелая аналитика, подключение LLM guardrails.

## 2. Что есть в старом проекте

Старый проект хорошо подходит как технический каркас, но не как доменная модель.

### Reuse

- `bot.py`
  - обвязка `maxapi`, polling, регистрация обработчиков, отправка сообщений;
- общая идея `handlers.py`
  - выделение бизнес-логики отдельно от точки входа;
- `config.py`
  - загрузка переменных окружения из `.env`;
- `state.py`
  - подход с локальным in-memory состоянием для MVP;
- общая схема callback-кнопок и маршрутизации через payload.

### Replace

- доменная модель в `models.py`
  - сейчас это `Purchase`, `Contract`, `SearchParams`;
- `gosplan_api.py`
  - полностью про закупки, к новому пилоту не относится;
- `gigachat_client.py`
  - заточен под парсинг закупочных запросов, не под кейсы;
- `formatters.py`
  - формирует карточки закупок и контрактов;
- большая часть старой бизнес-логики `handlers.py`
  - пагинация и детали для закупок не совпадают с ТЗ кейсов.

## 3. Главный вывод по проекту

Самый полезный перенос из старого бота — это не логика, а форма:

- MAX transport layer;
- inline-кнопки;
- event-driven структура;
- файл конфигурации;
- состояние пользователя в памяти.

Главный тормоз старого проекта — жесткая привязка запуска к чужим интеграциям:

- бот не стартовал без `GIGACHAT_CREDENTIALS`;
- предметная модель целиком про ГосПлан и закупки;
- текущий `handlers.py` слишком плотно связан с поиском закупок, чтобы его безопасно «допилить».

Поэтому правильная стратегия:

1. оставить MAX-каркас;
2. завести новый case-domain слой;
3. добавить локальные сиды и локальный поиск;
4. потом подключать SQLite, фото, аналитику и GigaChat MAX.

## 4. Целевая MVP-архитектура

### Уже выбранный каркас

- `bot.py`
  - только MAX transport и keyboard routing;
- `case_handlers.py`
  - сценарии диалога;
- `case_models.py`
  - `Case`, `CaseStep`, `CaseRun`, `CaseSubmission`, `CaseStepEvent`;
- `case_repository.py`
  - стартовые кейсы из ТЗ;
- `case_search.py`
  - локальный поиск по кейсам;
- `case_state.py`
  - состояние пользователя и черновики заявок;
- `case_formatters.py`
  - карточки, шаги, итог прохождения.

### Следующий слой

- `SQLite repository`
  - вместо in-memory state и каталога;
- `media storage`
  - путь к фото + привязка к шагам;
- `analytics events`
  - `search_started`, `case_selected`, `run_started`, `step_done`, `hint_requested`, `run_finished`, `new_case_created`;
- `admin view`
  - минимум список новых кейсов и экспорт;
- `gigachat_case_client`
  - нормализация запроса и fallback на похожий кейс.

## 6. Практический план следующих итераций

### Итерация 1

- Вынести каталог кейсов в `data/cases.json`
- Подключить SQLite
- Сохранить `case_runs`, `case_step_events`, `new_case_submissions`

### Итерация 2

- Добавить прием фото и комментариев
- Сохранять пути к фото
- Собрать итог прохождения из БД

### Итерация 3

- Подключить GigaChat MAX как слой нормализации запроса
- Оставить поиск гибридным: локальный индекс + LLM normalization
- Реализовать честный fallback «похожий кейс / создать новый»

### Итерация 4

- Добавить события аналитики
- Сделать минимальный просмотр пользовательских кейсов
- Подготовить smoke-checklist после каждого изменения

## 7. Риски и замечания

- Текущий стартовый поиск пока локальный и без LLM.
- Фото и комментарии пока отмечены как точки расширения, но еще не реализованы.
- История прохождения пока в памяти и пропадает после рестарта.
- Для реального пилота нужен переход на SQLite до начала тестирования с людьми.
- Токен бота уже заведен локально в `.env`, но после работ его стоит перевыпустить, потому что он был передан в чате.

## 8. Статус на сейчас

Сейчас проект уже переведен из логики «поиск закупок» в первый рабочий каркас case bot:

- бот стартует в MAX;
- `/start` и `/help` работают;
- есть главное меню;
- есть локальный подбор кейсов из стартового списка;
- есть карточка кейса;
- есть базовое прохождение по шагам;
- есть ветка создания нового кейса;
- старые закупочные модули остались в проекте как legacy-контур для референса.
